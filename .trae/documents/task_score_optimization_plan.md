# 任务系统得分优化实现计划

## 一、项目背景

当前任务系统的得分计算逻辑较为简单，主要基于完成时间的比例来计算得分。为了更好地激励用户按时完成任务，并提供更灵活的得分规则配置，需要对得分系统进行优化。

## 二、优化目标

1. **时间相关得分优化**：根据每个任务模板定制得分规则，包括超时惩罚和提前奖励
2. **审核相关得分优化**：审核时可以设置最终得分，优先级高于时间计算得分

## 三、详细实现计划

### [ ] 任务1：数据库表结构修改
- **Priority**：P0
- **Depends On**：None
- **Description**：
  - 修改 TaskTemplate 表，添加得分规则相关字段
  - 修改 Task 表，添加审核得分字段
  - 修改 TaskAudit 表，添加审核得分字段
- **Success Criteria**：
  - 数据库表结构修改完成，包含所有必要的得分规则字段
- **Test Requirements**：
  - `programmatic` TR-1.1：SQL语句执行成功，表结构修改正确
  - `programmatic` TR-1.2：现有数据不受影响
- **Notes**：
  - 需要添加的字段：
    - TaskTemplate：DelayPenaltyRule（超时惩罚规则）、MaxDelayPenalty（最大超时惩罚比例）、EarlyRewardRule（提前奖励规则）、MaxEarlyReward（最大提前奖励比例）
    - Task：AuditScore（审核得分）、FinalScore（最终得分）
    - TaskAudit：AuditScore（审核得分）

### [ ] 任务2：得分计算逻辑优化
- **Priority**：P0
- **Depends On**：任务1
- **Description**：
  - 优化 CalculateTaskScore 方法，支持根据模板的得分规则计算得分
  - 添加计算最终得分的方法，优先使用审核得分
- **Success Criteria**：
  - 得分计算逻辑正确实现，支持模板定制的得分规则
  - 最终得分计算优先使用审核得分
- **Test Requirements**：
  - `programmatic` TR-2.1：提前完成任务时，得分计算正确
  - `programmatic` TR-2.2：超时完成任务时，根据规则计算惩罚得分
  - `programmatic` TR-2.3：有审核得分时，最终得分使用审核得分
  - `programmatic` TR-2.4：无审核得分时，最终得分使用时间计算得分
- **Notes**：
  - 得分规则格式建议：JSON格式，包含时间区间和对应的得分调整规则
  - 示例：{"delayRules":[{"minutes":30,"penalty":5},{"minutes":60,"penalty":10}],"earlyRules":[{"minutes":30,"reward":3},{"minutes":60,"reward":6}]}
  - 格式说明：延迟X分钟，扣Y分；提前X分钟，加Y分
  - 注意：允许得分扣成负数

### [ ] 任务3：审核流程修改
- **Priority**：P0
- **Depends On**：任务1
- **Description**：
  - 修改 SaveTaskAudit 方法，支持审核时设置最终得分
  - 更新任务状态时，同时更新最终得分
- **Success Criteria**：
  - 审核时可以设置最终得分
  - 审核后任务的最终得分正确更新
- **Test Requirements**：
  - `programmatic` TR-3.1：审核时设置得分，任务的最终得分正确更新
  - `programmatic` TR-3.2：审核拒绝时，最终得分保持为0
  - `programmatic` TR-3.3：批量审核时，得分处理正确
- **Notes**：
  - 需要修改 SaveTaskAudit 方法的参数，添加审核得分字段
  - 需要更新相关的前端界面，支持审核时输入得分

### [ ] 任务4：前端界面修改
- **Priority**：P1
- **Depends On**：任务1
- **Description**：
  - 修改任务模板编辑界面，添加得分规则配置功能
  - 修改任务审核界面，添加得分输入功能
  - 更新任务列表界面，显示最终得分
- **Success Criteria**：
  - 任务模板编辑界面可以配置得分规则
  - 任务审核界面可以输入审核得分
  - 任务列表界面显示最终得分
- **Test Requirements**：
  - `human-judgement` TR-4.1：模板编辑界面得分规则配置功能易用
  - `human-judgement` TR-4.2：审核界面得分输入功能易用
  - `programmatic` TR-4.3：界面数据提交正确
- **Notes**：
  - 得分规则配置建议使用可视化界面，方便用户设置
  - 审核得分输入需要添加验证，确保得分在合理范围内

### [ ] 任务5：测试和验证
- **Priority**：P1
- **Depends On**：任务2、任务3、任务4
- **Description**：
  - 测试得分计算逻辑
  - 测试审核流程
  - 测试前端界面功能
- **Success Criteria**：
  - 所有功能正常工作
  - 得分计算准确
  - 界面操作流畅
- **Test Requirements**：
  - `programmatic` TR-5.1：所有功能测试通过
  - `human-judgement` TR-5.2：系统运行稳定，用户体验良好
- **Notes**：
  - 测试时需要覆盖各种场景，包括提前完成、按时完成、超时完成等
  - 需要测试审核通过和拒绝的情况

## 四、技术实现要点

1. **数据库设计**：
   - 使用 JSON 格式存储得分规则，提高灵活性
   - 添加必要的索引，提高查询性能

2. **得分计算**：
   - 优先使用审核得分作为最终得分
   - 无审核得分时，根据时间计算得分
   - 支持根据模板定制的得分规则

3. **审核流程**：
   - 审核时可以设置最终得分
   - 审核结果会影响最终得分

4. **前端界面**：
   - 提供直观的得分规则配置界面
   - 审核界面支持得分输入
   - 任务列表显示最终得分

## 五、预期效果

1. **更灵活的得分规则**：可以根据不同任务类型设置不同的得分规则
2. **更公平的得分计算**：审核得分优先级高于时间计算得分，确保得分的公正性
3. **更好的用户体验**：直观的界面操作，清晰的得分展示

## 六、风险评估

1. **数据迁移风险**：修改数据库表结构可能影响现有数据，需要谨慎处理
2. **兼容性风险**：修改得分计算逻辑可能影响现有任务的得分，需要确保向后兼容
3. **复杂度风险**：得分规则配置过于复杂可能增加用户使用难度，需要设计合理的界面

## 七、实施计划

1. **准备阶段**：分析需求，设计数据库表结构和得分计算逻辑
2. **开发阶段**：按照任务顺序实施开发
3. **测试阶段**：进行全面测试，确保功能正常
4. **部署阶段**：部署到生产环境，监控系统运行情况

## 八、总结

通过本次优化，任务系统的得分机制将更加灵活和公平，能够更好地激励用户按时完成任务，同时为审核人员提供更多的得分调整权限。这将有助于提高任务管理的效率和质量，为企业的任务管理提供更有力的支持。
